## 1. 현재 앱 구조

### 사용자 역할
| 역할 | 권한 | 접근 탭 |
|---|---|---|
| 원장 (owner) | 모든 기능 | 홈·재고·입출고·발주·관리 |
| 매니저 (manager) | 모든 기능 | 홈·재고·입출고·발주·관리 |
| 위생사 (hygienist) | 조회·입출고·발주요청 | 홈·재고·입출고·발주 |

### 데이터 구조
```
items     — 품목 (이름, 카테고리, 현재수량, 최소수량, 위치, 유통기한)
txs       — 입출고 이력 (품목, 유형, 수량, 메모, 담당자, 일시)
orders    — 발주 (품목, 수량, 요청자, 상태: pending→ordered→received/rejected)
surgeries — 수술 일정 (제목, 환자명, 날짜, 필요품목, 준비확인여부)
notifs    — 알림 (유형, 읽음여부, 관련품목/수술)
```

### 핵심 플로우
1. **로그인**: 사용자 선택 → PIN 4자리 → PBKDF2 해시 검증 → 12시간 세션
2. **입출고**: 재고 목록/바코드 스캔 → 수량 입력 → 즉시 반영 → 부족 시 알림
3. **발주**: 위생사 요청 → 장바구니 → 매니저 승인 → 입고 확인
4. **수술 준비**: 일정 등록 → 홈 카드 노출 → 품목 준비 확인 → 30분 전 리마인더
5. **알림**: 재고부족·발주상태·수술일정 자동 생성

## 2. 자동화 목표 프로세스

```
① 재고 부족 감지 (앱 자동)
       ↓
② 가격 비교 자동 조회 (백엔드 Playwright — 덴올·오스템몰·이덴트·메디세일)
       ↓
③ 앱 알림: "라텍스 장갑 부족 — 덴올 최저가 12,000원"
       ↓
④ 담당자 [주문 승인] 버튼 1번
       ↓
⑤ Playwright 자동 주문 실행 → 주문번호 저장
       ↓
⑥ 송장번호 자동 수집 (주문내역 페이지 스크래핑)
       ↓
⑦ 배송 상태 폴링 (1시간마다)
       ↓
⑧ "배달완료" 감지 → 앱 푸시 알림
       ↓
⑨ 담당자 [입고 확인] 버튼 1번 → 재고 자동 반영
```

**사람이 하는 일: 승인 1번 + 입고확인 1번**

### 자동화 가능 범위
| 단계 | 자동화 | 비고 |
|---|---|---|
| 재고 부족 감지 | ✅ 완전 자동 | 현재도 구현됨 |
| 가격 비교 | ✅ 완전 자동 | Playwright 스크래핑 |
| 주문 실행 | ⚠️ 반자동 | 사람이 최종 승인 후 실행 |
| 송장 수집 | ✅ 완전 자동 | 주문내역 스크래핑 |
| 배송 추적 | ✅ 완전 자동 | 스윗트래커 API |
| 입고 처리 | ⚠️ 반자동 | 알림 후 버튼 1번 (검수 필요) |

## 3. 기술 스택

### 현재
- React 19 PWA (Vite)
- localStorage (데이터 저장)
- @zxing/library (바코드 스캔)
- Web Push API (푸시 알림)

### 추가 필요
```
백엔드: Node.js + Express
자동화: Playwright (브라우저 자동화)
DB:     PostgreSQL 또는 Firebase Firestore
알림:   FCM (Firebase Cloud Messaging)
추적:   스윗트래커 API (배송 추적)
스케줄: node-cron (주기적 폴링)
암호화: AES-256-GCM + Key Vault (자격증명 보관)
```

### 추천 아키텍처
```
React PWA (Firebase Hosting)
      ↓
Firebase (Firestore + FCM + Auth)   ← DB·알림·인증
      ↓
맥미니 or 소형 EC2                  ← Playwright 전용 서버
  └ 스크래핑 + 자동주문 + 배송추적
```

## 5. SaaS 확장 구조

### 멀티테넌트 설계
```
각 치과 = 독립 테넌트
  ├── 고유 clinicId
  ├── 암호화된 도매 사이트 계정 (사이트별)
  ├── 결제 수단 (각자 법인카드/후불)
  └── 독립 Playwright 세션
```

### 세금계산서 문제
- **없음** — 각 치과 본인 계정으로 주문하므로 세금계산서는 치과로 자동 발행
- 치과 B2B 도매는 대부분 후불 세금계산서 방식 → 결제 자동화 불필요

### 온보딩 프로세스
```
1. 사용하는 도매 사이트 선택
2. 각 사이트 ID/PW 입력 (암호화 저장)
3. 결제 수단 설정 (후불 or 저장 카드)
4. 품목별 자동발주 on/off 설정
5. 최대 주문금액 한도 설정
```

### 수익 모델
| 방식 | 내용 |
|---|---|
| 월정액 SaaS | 치과당 월 3~5만원 |
| 건당 수수료 | 자동주문 성공 시 건당 500~1,000원 |
| 혼합 | 기본료 + 건당 |

```
치과 10개: 매출 30만원 - 비용 5만원 = 순이익 25만원
치과 30개: 매출 90만원 - 비용 12만원 = 순이익 78만원
```

## 6. 보안 체크리스트

### 🔴 Critical (반드시)
- [ ] 도매 사이트 PW — AES-256-GCM 암호화 + Key Vault 분리 보관
- [ ] 자동 주문 전 담당자 승인 단계 필수
- [ ] 1회 최대 주문금액 한도 설정
- [ ] 멀티테넌트 데이터 격리 (clinicId 검증)
- [ ] 법인카드 번호 직접 저장 금지 (사이트 저장 카드 활용)

### 🟡 High
- [ ] 관리자 계정 2FA (OTP 추가)
- [ ] 개인정보보호법 — 환자명 처리방침 + 위탁계약서
- [ ] API Rate Limiting
- [ ] 로그에 PW·카드번호 미기록

### 🟢 Medium
- [ ] 맥미니 FileVault 디스크 암호화
- [ ] UPS (무정전 전원장치)
- [ ] 자동 백업 (Backblaze B2 등)
- [ ] Playwright 요청 간 랜덤 딜레이 (봇 감지 우회)

## 7. 화면 변화 (자동화 적용 후)

### 발주 탭 개편
```
기존: 장바구니 → 일괄 발주요청
변경: 자동대기 → 진행중 → 완료 파이프라인
```

### 새로 추가되는 화면
- **가격 비교 시트** — 사이트별 가격·재고·배송일 비교
- **배송 추적 상세** — 송장번호·배송단계·입고확인 버튼
- **도매 설정 탭** — 사이트 계정 연결·자동발주 설정

### 알림 유형 추가
```
기존: 재고부족·발주상태·수술일정
추가: 가격비교완료·주문승인필요·주문완료
      송장등록·배송출발·배달완료·주문실패
```
